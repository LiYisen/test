# 期货回测系统需求文档 (Requirements)

## 1. 项目概述
本项目是一个基于 Go 语言和 Python (akshare) 协同开发的期货回测系统。其核心目的是验证"阴阳线突破"交易策略在不同期货品种和时间范围内的表现。系统采用**信号生成与资金计算分离**的架构，信号层可直接切换到实盘交易，资金层作为独立模块接收信号进行回测分析。

## 2. 功能需求

### 2.1 数据管理 (Data Management)
- **数据源**: 通过 Python `akshare` 库获取新浪财经的期货日 K 线数据。
- **品种支持**: 支持大商所、郑商所、上期所、中金所的所有主流期货品种（如 RB, IF, CF 等）。
- **主力合约识别**: 
    - 自动生成回测区间内所有可能存在的合约代码（如 RB2601 - RB2712）。
    - **初始主力合约**：回测第一天，选择持仓量最大的合约作为初始主力合约。
    - **单向切换**：合约只往后续月份切换，不往回切换。例如当前主力是2601合约，新的主力只能是2602、2603及之后的合约，不能是2512及之前的合约。
    - **切换条件**：当后续月份合约的持仓量和成交量**同时**超过当前主力合约时，切换为新的主力合约。
    - **保持机制**：如果后续月份合约只有一项指标（持仓量或成交量）超过当前主力合约，则保持当前主力合约不变。
    - 识别过程需具备高性能，复用已获取的 K 线数据。
- **交易日历**: 自动从数据源获取真实的交易日历。

### 2.2 交易策略 (Strategy - YinYang)

#### 2.2.1 阴阳线元素合成 (Aggregation)
- **阳线元素 (Yang1)**: 连续阳线的最高点与最低点的合并区间。
- **阴线元素 (Yin1)**: 连续阴线的最高点与最低点的合并区间。
- **降级机制**: 当趋势反转时，旧的 Yang1/Yin1 降级为 Yang2/Yin2，新 K 线成为新的 Yang1/Yin1。
- **十字星处理**: 当 K 线收盘价等于开盘价时，沿用上一个交易日的方向（就近合入）。

#### 2.2.2 信号触发 (Signals)
- **无持仓**: 突破最近一个阳线元素和阴线元素的最高点之最大值做多；突破其最低点之最小值做空。
- **持仓中**: 动态跟踪反向止损/反手信号价。

#### 2.2.3 临时阴阳集合 (Temporary Yin-Yang Collection)
当在一根 K 线上触发信号价格产生交易时（移仓换月导致的交易不算），需要生成临时阴阳集合：

**触发条件**：
| 场景 | K线类型 | 是否触发临时集合 |
|------|--------|----------------|
| 做多 | 阴线 | ✅ 触发 |
| 做多 | 阳线 | ❌ 不触发 |
| 做多 | 十字星 | ❌ 不触发 |
| 做空 | 阳线 | ✅ 触发 |
| 做空 | 阴线 | ❌ 不触发 |
| 做空 | 十字星 | ❌ 不触发 |

**生成逻辑**：
- **做多 + 阴线**: 把阴线当成阳线处理。就近的是阴1，则自己成为阳1；就近的是阳1，就合入阳1。
- **做空 + 阳线**: 把阳线当成阴线处理。就近的是阳1，则自己成为阴1；就近的是阴1，就合入阴1。

**使用规则**：
- 临时阴阳集合只影响下一根 K 线的信号价格计算。
- 下一根 K 线结束后，临时阴阳集合自动作废（无论是否成交）。
- 临时阴阳集合不影响正常阴阳集合的状态。

**临时阴阳集合的状态方向（关键）**：
临时阴阳集合有自己的状态方向（IsYang），与当前K线的实际方向不同：

| 场景 | 当前K线实际方向 | 临时集合状态方向（IsYang） |
|------|----------------|--------------------------|
| 做多 + 阴线 | 阴线（false） | 阳线（true） |
| 做空 + 阳线 | 阳线（true） | 阴线（false） |

**信号价格计算时**：
- 如果存在临时阴阳集合，使用临时集合的状态方向（IsYang）
- 如果不存在临时阴阳集合，使用当前K线的实际方向

#### 2.2.4 时序逻辑
**核心原则**: 严格遵循时序，禁止使用未来函数。

**每日处理流程**：
```
输入: KLine_T (第T天的K线数据)
1. 如果存在临时阴阳集合（由K线T-1生成）：
   - 使用临时集合的状态方向计算信号价格
   - 用新计算的信号价格进行触发判断
2. 触发判断完成后：
   - 标记临时集合为已使用
3. 更新阴阳线状态（使用KLine_T的数据）
4. 如果成交：
   - 记录交易信号
   - 根据条件生成临时阴阳集合
   - 计算反向信号价（用于下一根K线）
5. 如果不成交：
   - 更新信号价（用于下一根K线）
```

**重要**：临时集合必须先用于计算信号价格，然后才能标记为已使用。

**时序关键点**：
- 信号价格必须在上一根K线结束后计算完成，用于当前K线的触发判断
- 阴阳集合状态更新发生在触发判断之后
- 临时阴阳集合生成发生在成交之后，影响下一根K线

#### 2.2.5 无持仓信号价格计算（重要）

**区分两种无持仓情况**：

**情况1：真正的无持仓（回测开始，从未开过仓）**
- 条件：hasEverHeldPosition = false
- 计算方式：只使用阴1阳1，不使用阴2阳2
```
做多信号价 = max(阴1.High, 阳1.High)
做空信号价 = min(阴1.Low, 阳1.Low)
```

**情况2：有持仓历史但当前无持仓**
- 条件：hasEverHeldPosition = true
- 计算方式：根据当前K线方向使用阴2阳2
```
如果当前K线为阳线：
  做多信号价 = max(阴1.High, 阳1.High)
  做空信号价 = min(阳1.Low, 阴2.Low)  // 使用阴2

如果当前K线为阴线：
  做多信号价 = max(阴1.High, 阳2.High)  // 使用阳2
  做空信号价 = min(阴1.Low, 阳1.Low)
```

**解释**：一旦开过仓，就进入正常逻辑，需要使用阴1阳1和阴2阳2。

#### 2.2.6 杠杆计算
杠杆系数代表账户承担的百分比风险（例如 `-leverage 4` 代表 4% 风险）。

**计算公式**：
- `杠杆 = (风险系数 / 100) * 开仓价格 / (开仓价格 - 止损价)`
- 硬限制：最大杠杆限制为 6.0 倍

**止损价定义**：
| 开仓方向 | 止损价计算 | 说明 |
|---------|-----------|------|
| 开多 | min(阴1.Low, 阳1.Low) | 使用阴1阳1的低点作为止损价 |
| 开空 | max(阴1.High, 阳1.High) | 使用阴1阳1的高点作为止损价 |

**注意**：止损价在开仓时确定，使用开仓时刻的阴阳集合状态。与持仓期间的"反向信号价"不同，反向信号价会根据K线类型和阴2阳2动态更新。

#### 2.2.7 反向信号价计算
**持有多头时，做空信号价更新**：
| 当前K线类型 | 做空信号价计算 |
|------------|---------------|
| 阳线 | min(阳1.Low, 阴1.Low) |
| 阴线 | min(阳1.Low, 阴2.Low) |

**持有空头时，做多信号价更新**：
| 当前K线类型 | 做多信号价计算 |
|------------|---------------|
| 阳线 | max(阴1.High, 阳2.High) |
| 阴线 | max(阴1.High, 阳1.High) |

### 2.3 信号与资金分离 (Signal-Fund Separation)
- **信号层 (Signal Layer)**:
    - 策略只负责生成交易信号，不涉及资金计算。
    - 持仓信息使用 `SignalPosition`（仅包含合约、方向、开仓价、开仓日期、杠杆）。
    - 信号层可直接对接实盘交易系统。
- **资金层 (Fund Layer)**:
    - 接收交易信号作为输入，独立进行资金计算。
    - 持仓信息使用 `Position`（包含数量、当前价格等资金相关字段）。
    - 负责保证金计算、盈亏计算、每日资金曲线绘制、各类指标统计。

### 2.4 移仓换月 (Rollover)

#### 2.4.1 延迟执行机制
- **T日**: 检测到主力合约切换，记录待移仓信息
- **T+1日**: 在开盘价执行移仓换月

**时序要求**：
- 主力合约切换的判断使用T日收盘后的数据
- 移仓换月操作在T+1日开盘时执行
- 禁止在T日就执行移仓（避免使用未来函数）

#### 2.4.2 执行逻辑
- 平旧仓：使用旧主力合约T+1日开盘价
- 开新仓：使用新主力合约T+1日开盘价

#### 2.4.3 新合约开仓方向确定
**模拟交易法**：
1. 获取新合约从上市首日到移仓前一日的所有历史K线
2. 使用策略模拟在新合约上连续交易
3. 以模拟结果确定的持仓方向作为新合约开仓方向

**优势**：
- 新合约方向独立于旧合约
- 方向可能与旧合约相反
- 符合策略逻辑的一致性

#### 2.4.4 多合约状态管理
- 每个合约维护独立的阴阳集合状态
- 系统初始化时预加载所有合约的历史K线，构建完整的阴阳集合
- 移仓换月时新合约的阴阳集合状态是完整的，可直接用于信号价计算
- 主力合约切换时，立即切换当前合约标识

#### 2.4.5 临时集合排除
移仓换月导致的交易不触发临时阴阳集合。

### 2.5 结果输出 (Reporting)
- **交易信号列表**: 日期、合约、方向、成交价、杠杆、信号类型。
- **阴阳线状态历史**: 输出每日的阴阳线元素极值及对应的双向触发信号价。
- **主力切换信息**: 显示切换日期、旧合约和新合约的完整阴阳集合状态及信号价格。
- **临时阴阳集合记录**: 单独输出触发临时阴阳集合的记录，包含临时阳1、临时阴1、临时阳2、临时阴2。
- **资金计算（独立模块）**: 每日账户记录、统计摘要等，由资金层独立计算和输出。

### 2.6 回测规则 (Rules)
- 支持规则化的回测约束（如最大回撤限制、杠杆限制、资金下限、持仓上限等）。
- 规则引擎在信号生成后、资金计算前进行校验。

## 3. 时序性要求（关键）

### 3.1 数据获取时序
- K线数据按日期升序排列
- 只使用当前日期及之前的数据进行计算
- 禁止预读取未来日期的数据

### 3.2 状态更新时序
```
日期T的处理顺序：
1. 读取KLine_T
2. 使用T-1日结束时计算的信号价格判断触发
3. 更新阴阳集合状态（使用KLine_T数据）
4. 计算T日结束后的信号价格（用于T+1日）
```

### 3.3 移仓换月时序
```
T日收盘后：
1. 检测主力合约是否切换
2. 如果切换，记录待移仓信息
3. 更新所有合约的阴阳集合状态

T+1日开盘：
1. 执行移仓换月交易
2. 平旧合约仓位
3. 开新合约仓位（方向由模拟交易确定）
4. 继续处理T+1日的正常交易逻辑
```

### 3.4 多合约状态时序
- 所有合约的阴阳集合状态每日同步更新
- 只有当前主力合约参与交易判断
- 非主力合约的状态更新用于移仓换月时的状态完整性

## 4. Web界面需求

### 4.1 整体布局
- 系统采用**单页应用（SPA）**架构，所有功能集成在一个页面中。
- 页面左侧为**固定导航栏**，右侧为内容区域。
- 导航栏支持**收起/展开**，在窄屏下自动折叠为图标模式。

### 4.2 导航栏
- 导航栏固定在页面左侧，包含以下功能模块入口：
  - **回测**: 单品种策略回测
  - **组合分析**: 多品种组合收益分析
  - **基金**: 多品种基金模式回测
- 当前选中模块具有明确的视觉标识（高亮边框、颜色变化）。
- 导航栏底部提供"收起"按钮，点击后导航栏折叠为图标模式。

### 4.3 页面切换
- 点击导航栏选项时，通过客户端 JavaScript 实现**无刷新切换**。
- 切换过程流畅，无页面重载。
- 切换到对应模块时自动加载该模块所需数据。

### 4.4 回测模块
- 提供品种搜索选择器，支持按代码、名称、拼音搜索。
- 提供日期选择器、策略选择器及策略参数配置。
- 回测完成后展示统计指标卡片、资金曲线与回撤图表、日收益图表、交易记录表格。
- 支持查看和删除历史回测结果。

### 4.5 组合分析模块
- 以多选列表方式展示已有回测结果，支持勾选多个结果。
- 至少选择2个时间段有重叠的结果后可执行组合分析。
- 展示组合统计指标、组合资金曲线与回撤、组合日收益、各品种净值对比图表。

### 4.6 基金模块
- **基金回测**: 选择已配置的基金，设定日期范围执行回测，展示统计指标和净值曲线图表。
- **回测结果**: 列出历史基金回测结果，支持查看和删除。
- **基金配置**: 创建新基金，配置持仓品种、策略、权重和参数，权重总和必须为1.00。

### 4.7 响应式设计
- 在768px以下屏幕宽度，导航栏自动折叠为图标模式。
- 表单布局从多列变为单列。
- 统计卡片从自适应列数变为2列。

### 4.8 错误处理
- 数据加载失败时显示友好的错误提示信息。
- API请求异常时在页面内展示错误消息，不使用浏览器原生弹窗（删除确认除外）。
- 图表数据为空时显示"暂无数据"提示。

## 5. 基金模式需求

### 5.1 基金配置
- 支持创建多个基金，每个基金包含唯一ID、名称、描述。
- 每个基金可配置多个持仓品种，每个品种指定：
  - 品种代码
  - 策略名称
  - 权重（所有品种权重之和必须为1.00）
  - 策略参数（如杠杆系数）
- 基金配置保存在 `config/funds.json` 文件中。

### 5.2 基金回测
- 基金回测基于**净值**计算，不依赖初始资金。
- 并发执行各品种的独立回测，通过通道收集结果。
- 各品种的回测结果按权重加权合并为基金整体净值曲线。
- 回测结果保存在 `ret/funding/{fund_id}/{result_id}/` 目录下。

### 5.3 结果存储
- 基金回测结果独立存储，不与单品种回测结果混在一起。
- 存储内容包括：基金每日净值记录、各品种表现汇总、统计指标。
- 支持查看历史回测结果和删除。

### 5.4 统计指标
- 总收益率、年化收益率
- 最大回撤、最大回撤比例
- 夏普比率、卡玛比率
- 胜率、交易天数、总交易次数

## 6. 非功能需求
- **性能**: 主力合约识别和信号计算应在数秒内完成。
- **鲁棒性**: 自动捕获并跳过无效或未上市的合约请求。
- **编码**: 在 Windows 环境下，终端输出需支持 UTF-8 编码，避免中文乱码。
- **可扩展性**: 信号层与资金层解耦，信号层可直接切换到实盘交易。
- **并发安全**: 基金回测的并发执行使用通道而非共享切片，配置加载使用 `sync.Once` 保证线程安全。
- **精度**: 财务计算使用 `float64` 类型，回测阶段精度满足需求。
